Investigate-our-usage → shipped a whole feature: opt-in usage-coaching hooks
(ADR 0001), released as v0.36.0. Started as "we used a lot of sessions, look at
our usage" and turned into design → build → release → live-install.

## What happened
1. **Usage investigation.** Parsed the real Claude Code transcripts (`~/.claude/
   projects/**/*.jsonl`) directly: **~$50.6k API-equivalent over 7 days**, ~68.6k
   requests, and **~79% of the cost was cache-read** (26.75B tokens) — i.e. large
   context carried across many turns of long sessions, not output volume. Biggest
   sessions ran **7,000–9,300 turns / ~$2,400 cache-read each**.
2. **tokenops's own data was stale.** `tokenops_data_sources`/`top_consumers` showed
   only ~7.9B opus tokens/30d with **cost $0** — because (a) no provider configured
   (`state: degraded`, `providers_unconfigured`) so cost couldn't be priced, and
   (b) the `claude_code_jsonl` vendor-usage poller had ingested nothing since
   ~June 30 (0 events/24h). Fixed the provider (`provider set anthropic …`).
3. **Version skew.** The RUNNING tokenops MCP server reported **0.25.1** while the
   CLI/homebrew was **0.30.1** and the repo was **0.35.0**. Root cause: the MCP
   `serve` process was a long-lived stdio process started *before* a `brew upgrade`
   — the on-disk binary + both MCP configs already pointed at homebrew 0.30.1, so
   it was purely a stale running process. Killed the stale `tokenops serve`
   processes; user `/mcp`-reconnected. `brew upgrade` 0.30.1→0.35.0→(later)0.36.0.
4. **ADR 0001 (Accepted)** — opt-in, hook-driven, threshold/budget-based coaching
   layer that makes the retrospective `coach`/`scorecard` engines *live*. Star
   control: a Stop-hook that nudges to `/compact`.
5. **Phase 1 (#127):** `coach-hook` Claude Code **Stop** hook (reads the transcript
   TAIL, ~256 KiB, never the whole file) + `tokenops hooks install/uninstall/status`
   (idempotent settings.json merge, `.bak` backup, atomic write, refuses malformed
   JSON, version-mismatch warn). Mirrors the `read-guard` pattern; **fail-open**.
6. **Phase 1.1 (#128):** replaced the flat per-turn threshold with a **cumulative
   per-session $-budget + graduated latched alerts** (50/75/100% + over-budget
   escalation 200/300…), default **$50**, `--budget`. Fires only the single highest
   newly-crossed tier per Stop. Dogfooding proved the per-turn model was wrong.
7. **Released v0.36.0 (#129 + tag)**, `brew upgrade`, installed BOTH hooks on the
   homebrew binary. Pruned 3 stale agent worktrees.

## Decisions Made
- **Cumulative $-budget over per-turn threshold** — the live test showed this very
  session runs ~609k cache-read/turn (under a 1M per-turn line) yet cost >$1,800
  cumulatively. A per-turn threshold fundamentally misses long-*flat* sessions;
  a GitHub-Actions-style cumulative budget catches them. User's idea; data-grounded
  default $50 from the real per-session distribution (median $107, max $2,397).
- **$-normalized = model-agnostic** — budgeting in dollars (via `spend.DefaultTable`)
  means fable/haiku sessions accrue slower against the same tiers. Answers "for 1M
  but also for models under 1M."
- **Fail-open is sacred** for Claude Code hooks — any error → exit 0, no output; the
  Stop hook emits `{"systemMessage":…,"suppressOutput":true}` and NEVER
  `decision:"block"` (which would force the agent to keep going).
- Reused the existing pricing catalog (cache-write priced at input rate — no distinct
  cache-write field in the `spend` table; cache-read dominates so negligible).

## What Went Wrong
- **`go install` dev build shadowed homebrew on PATH.** After merging, I built the
  binary to `~/go/bin/tokenops` to dogfood pre-release; that shadowed
  `/opt/homebrew/bin/tokenops`, so `hooks install` wired the hooks (and `version`)
  to the DEV build even after `brew upgrade`. Fixed by installing via the explicit
  homebrew path then `rm`-ing the dev build. LESSON: install hooks via an explicit
  stable binary path, and remove `go install` artifacts after a release.
- **MCP servers are long-lived stdio processes** — `brew upgrade` does NOT restart
  them; you must `/mcp` reconnect (or restart the host) to pick up the new binary.
  This is the mechanism behind the whole version-skew confusion.
- **tokenops undercounted its own usage** — the direct transcript parse showed ~27B
  opus tokens/7d vs the MCP store's 7.9B/30d, because ingestion stalled + cost was
  $0. The tool that measures usage was itself not being fed. Worth a health signal.
