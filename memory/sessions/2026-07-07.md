Investigate-our-usage → shipped a whole feature: opt-in usage-coaching hooks
(ADR 0001), released as v0.36.0. Started as "we used a lot of sessions, look at
our usage" and turned into design → build → release → live-install.

## What happened
1. **Usage investigation.** Parsed the real Claude Code transcripts (`~/.claude/
   projects/**/*.jsonl`) directly: **~$50.6k API-equivalent over 7 days**, ~68.6k
   requests, and **~79% of the cost was cache-read** (26.75B tokens) — i.e. large
   context carried across many turns of long sessions, not output volume. Biggest
   sessions ran **7,000–9,300 turns / ~$2,400 cache-read each**.
2. **tokenops's own data was stale.** `tokenops_data_sources`/`top_consumers` showed
   only ~7.9B opus tokens/30d with **cost $0** — because (a) no provider configured
   (`state: degraded`, `providers_unconfigured`) so cost couldn't be priced, and
   (b) the `claude_code_jsonl` vendor-usage poller had ingested nothing since
   ~June 30 (0 events/24h). Fixed the provider (`provider set anthropic …`).
3. **Version skew.** The RUNNING tokenops MCP server reported **0.25.1** while the
   CLI/homebrew was **0.30.1** and the repo was **0.35.0**. Root cause: the MCP
   `serve` process was a long-lived stdio process started *before* a `brew upgrade`
   — the on-disk binary + both MCP configs already pointed at homebrew 0.30.1, so
   it was purely a stale running process. Killed the stale `tokenops serve`
   processes; user `/mcp`-reconnected. `brew upgrade` 0.30.1→0.35.0→(later)0.36.0.
4. **ADR 0001 (Accepted)** — opt-in, hook-driven, threshold/budget-based coaching
   layer that makes the retrospective `coach`/`scorecard` engines *live*. Star
   control: a Stop-hook that nudges to `/compact`.
5. **Phase 1 (#127):** `coach-hook` Claude Code **Stop** hook (reads the transcript
   TAIL, ~256 KiB, never the whole file) + `tokenops hooks install/uninstall/status`
   (idempotent settings.json merge, `.bak` backup, atomic write, refuses malformed
   JSON, version-mismatch warn). Mirrors the `read-guard` pattern; **fail-open**.
6. **Phase 1.1 (#128):** replaced the flat per-turn threshold with a **cumulative
   per-session $-budget + graduated latched alerts** (50/75/100% + over-budget
   escalation 200/300…), default **$50**, `--budget`. Fires only the single highest
   newly-crossed tier per Stop. Dogfooding proved the per-turn model was wrong.
7. **Released v0.36.0 (#129 + tag)**, `brew upgrade`, installed BOTH hooks on the
   homebrew binary. Pruned 3 stale agent worktrees.

## Decisions Made
- **Cumulative $-budget over per-turn threshold** — the live test showed this very
  session runs ~609k cache-read/turn (under a 1M per-turn line) yet cost >$1,800
  cumulatively. A per-turn threshold fundamentally misses long-*flat* sessions;
  a GitHub-Actions-style cumulative budget catches them. User's idea; data-grounded
  default $50 from the real per-session distribution (median $107, max $2,397).
- **$-normalized = model-agnostic** — budgeting in dollars (via `spend.DefaultTable`)
  means fable/haiku sessions accrue slower against the same tiers. Answers "for 1M
  but also for models under 1M."
- **Fail-open is sacred** for Claude Code hooks — any error → exit 0, no output; the
  Stop hook emits `{"systemMessage":…,"suppressOutput":true}` and NEVER
  `decision:"block"` (which would force the agent to keep going).
- Reused the existing pricing catalog (cache-write priced at input rate — no distinct
  cache-write field in the `spend` table; cache-read dominates so negligible).

## What Went Wrong
- **`go install` dev build shadowed homebrew on PATH.** After merging, I built the
  binary to `~/go/bin/tokenops` to dogfood pre-release; that shadowed
  `/opt/homebrew/bin/tokenops`, so `hooks install` wired the hooks (and `version`)
  to the DEV build even after `brew upgrade`. Fixed by installing via the explicit
  homebrew path then `rm`-ing the dev build. LESSON: install hooks via an explicit
  stable binary path, and remove `go install` artifacts after a release.
- **MCP servers are long-lived stdio processes** — `brew upgrade` does NOT restart
  them; you must `/mcp` reconnect (or restart the host) to pick up the new binary.
  This is the mechanism behind the whole version-skew confusion.
- **tokenops undercounted its own usage** — the direct transcript parse showed ~27B
  opus tokens/7d vs the MCP store's 7.9B/30d, because ingestion stalled + cost was
  $0. The tool that measures usage was itself not being fed. Worth a health signal.

## Session 2 — stale-ingestion health warning → v0.37.0

Follow-up polish on the "tokenops undercounted its own usage" open thread from
Session 1: closed it with a real feature + release.

- **Stale-ingestion health warning (#131).** `tokenops status` (CLI + the
  `tokenops_status` MCP tool) now flags RUNTIME ingestion staleness, not just
  config blockers: an ENABLED vendor-usage source with **0 events in 48h** →
  a soft `warnings` entry + remediation `next_action`, degrading `state`
  ready→degraded (keeps `ready:true`). Stale sources go in `warnings`, never
  `blockers`. New `config.CheckStaleIngestion` (store-interface-based, nil-safe)
  + a shared `VendorUsageSources()`/`EnabledVendorUsageSources()` helper that
  vendor-usage status was refactored to reuse (single source of truth). 48h const.
- **Released v0.37.0 (#132)** — CHANGELOG promotion + open-thread resolution in
  one PR; tag → goreleaser → homebrew tap; `brew upgrade` 0.36.0→0.37.0.
- **Verified live.** Started the daemon, ran `tokenops status`: it flagged
  `anthropic-cookie` (enabled, 0/48h) but NOT `claude_code_jsonl` (fresh over 48h
  after the /mcp reconnect resumed its poller). Precise — distinguishes "resumed"
  from "still dead," no false positive.

## Decisions Made (Session 2)
- **Ingestion staleness is a `warnings` signal, NOT a `blocker`.** Blockers are
  config-level hard gates (providers_unconfigured etc.); a stalled poller is a
  soft runtime signal — surface it, degrade state, but keep serving (`ready:true`).
- **48h window** (not 24h like vendor-usage status) to avoid nagging when a vendor
  simply wasn't used for a day. Honest caveat documented: 0 events can also mean
  legitimate non-use.
- **Verify via the online/MCP path** — the CLI `status` OFFLINE path (daemon down)
  omits warnings (can't reach the store); the warning renders on the daemon-online
  CLI path and the MCP `tokenops_status` tool.

## What Went Wrong (Session 2)
- First stale-ingestion agent run was stopped mid-flight (user interrupt); it had
  committed nothing (worktree at main HEAD) — removed the empty worktree, relaunched
  clean. No salvage needed.

## Session 3 — Opus pricing fix + pricing research (ADR 0002) → v0.38.0

Started from "what's fable-5's rate?" → found the whole pricing catalog was
hand-maintained + drifting → fixed the immediate bug AND built the systemic
prevention.

- **Opus 4.x pricing was ⅓ too low (#135).** `pricing.yaml` priced
  claude-opus-4-{8,7,6} at `$5/$25/$0.50` per M — exactly ⅓ of standard Opus
  (`$15/$75/$1.50`). Cache-read `$0.50` was internally consistent with the wrong
  `$5` input, so nothing flagged it; every Opus API-equivalent cost (spend, the
  coach-hook budget) was understated **3×**. Fixed the full row. Verified the rest
  of the Anthropic block is consistent (cache = 10% of input, output = 5×) — Opus
  was the ONLY miscalibration. Big test ripple: 5 tests fixed by me (incl. the
  router-savings test, which had routed fable→opus *only because opus was
  miscosted cheap* — switched to opus→sonnet, a real cost-down) + 8 coachhook
  infra tests recalibrated (budgets scaled 3× so tier/nudge decisions unchanged).
- **fable-5** ($10/$50/$1.00) is internally consistent but its ABSOLUTE rate is
  unverified (like opus was — consistency ≠ correctness). Left as-is pending the
  operator's confirmation / the LiteLLM refresh.
- **ADR 0002 (Accepted): researched, effective-dated pricing snapshots.** Built
  BOTH phases:
  - **Phase 1 (#137):** `internal/contexts/spend/pricing` — pluggable `Source`
    (default LiteLLM `model_prices_and_context_window.json`), timestamped+sourced
    `Snapshot`s under `~/.tokenops/pricing/`, embedded `pricing.yaml` as the
    offline baseline, `tokenops pricing refresh/show/diff/lint`. **refresh diffs**
    (the opus ⅓ would've shouted `cache_read 0.5 → 1.5 (+200%)`); **consistency
    guard** = the cache≈10%/output≈5× check that caught opus, run as a lint.
  - **Phase 2 (#138):** effective-dated cost engine — each event priced at the
    rate in effect at its timestamp (`spend.NewDatedEngine`/`ComputeAt`; snapshot→
    dated-table conversion in `pricing`). No import cycle (spend must NOT import
    pricing; primitive keyed by time.Time in spend, conversion in pricing). Note:
    `PromptEvent` has no Timestamp — it's on the `Envelope`; threaded via
    `ComputeAt`/`ApplyToEnvelope`. Backward-compatible (baseline-only == today).
- **Released v0.38.0 (#139)** + brew upgrade + **verified live**: `pricing show`
  reads opus 15/75/1.5; `pricing lint` clean (0 anomalies); the coach-hook now
  prices at $1.50/M. The one un-verified path is the LIVE `pricing refresh` (the
  sandbox blocks outbound network) — handed to the operator to run.

## Decisions Made (Session 3)
- **Opus row → $15/$75/$1.50** (standard Opus 4.x). Cache-read can't be fixed in
  isolation (it's 10% of input); the whole row was ⅓, so all of input/output/cache
  moved together.
- **Pricing must be researched, not hand-maintained** (ADR 0002). Source = LiteLLM
  JSON (pluggable). Snapshots are timestamped + sourced; cost prices at the rate in
  effect at the event's time (effective-dating). Refresh **diffs** so drift is loud
  (the exact failure mode that hid the opus ⅓). A consistency guard lints every
  snapshot to defend against the source itself being wrong.
- **spend must not import pricing** — the effective-dating primitive lives in
  spend (keyed by time.Time), the snapshot→table conversion in pricing.

## What Went Wrong (Session 3)
- Consistency ≠ correctness: opus was internally consistent at the WRONG ⅓, so no
  guard caught it — a human noticing the cache-read line looked low did. The guard
  now runs, but the deeper fix is sourcing (ADR 0002).

## Session 4 — the Opus "fix" was WRONG; LiteLLM caught it (v0.39.0)

CORRECTION to Session 3: my Opus "correction" was itself the error. Ran the real
`tokenops pricing refresh` (network works from the machine — the sandbox caution
was wrong) and LiteLLM disagreed with our baseline:
`~ claude-opus-4-8 input 15 → 5`. Checked the raw feed:
**Anthropic CUT Opus pricing at 4.5** — Opus 4 / 4.1 = $15/$75/$1.50, but
**Opus 4.5–4.8 = $5/$25/$0.50** (the value the catalog ORIGINALLY had). So $0.50
cache-read was correct all along; I anchored on old-Opus $15 and "fixed" a right
value.

- **Reverted #135 (v0.39.0, #141/#142)** — Opus back to $5/$25/$0.50, original
  tests restored, Phase-1 baseline/fixture assertions corrected to $5. `git revert`
  was clean.
- **Adopted the LiteLLM snapshot** — `pricing refresh` written to
  `~/.tokenops/pricing/snapshots/`; the Opus diff is now CLEAN (baseline matches
  source). `pricing lint` → 17 models, no anomalies. coach-hook now prices at
  $0.50 ("$17+" where the wrong $1.50 showed "$70+").
- **My usage analyses were 3× too high**: ~$50.6k/7d → ~$17k; today ~$1,362 → ~$450;
  the $2,400 sessions → ~$800. tokenops's original rate was right; my external math
  (using $15) was wrong.

LESSON (the real one): the **consistency guard cannot catch this** — $5 and $15 are
both internally consistent (cache=10%, output=5×). Only the SOURCE can. This is the
entire reason ADR 0002 exists, and it caught a confidently-wrong human on first use.
